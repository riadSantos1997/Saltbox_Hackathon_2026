# A3 · Four Permission-Category Scrapers

## 🎯 Objective
Implement the four scrape functions (Object Settings, System Permissions, App Permissions, Apex Class Access). Run both orgs in parallel via `Promise.all()`. Return typed `ScrapeResult[]` matching the contract in `types.ts`. Implements **F-02**.

## 📂 Files
- `lib/salesforce/queries.ts` — SOQL builders for each category
- `lib/salesforce/scrape.ts` — `scrape(category, profileName, orgIds): Promise<ScrapeResult[]>`
- `app/api/salesforce/scrape/route.ts` — HTTP wrapper

## ✅ Acceptance Criteria
- [ ] **Object Settings:** `SELECT SobjectType, PermissionsCreate, PermissionsRead, PermissionsEdit, PermissionsDelete, PermissionsViewAllRecords, PermissionsModifyAllRecords FROM ObjectPermissions WHERE Parent.Profile.Name = '{name}'` — optionally filter by selected object API names.
- [ ] **System Permissions:** `SELECT Id, PermissionsModifyAllData, ... FROM PermissionSet WHERE IsOwnedByProfile = true AND Profile.Name = '{name}'` — boolean flags flattened to rows.
- [ ] **App Permissions:** `SELECT SetupEntityId, SetupEntityType FROM SetupEntityAccess WHERE Parent.Profile.Name = '{name}' AND SetupEntityType = 'TabSet'` — joined with TabSet metadata.
- [ ] **Apex Class Access:** `SELECT SetupEntityId FROM SetupEntityAccess WHERE Parent.Profile.Name = '{name}' AND SetupEntityType = 'ApexClass'` — joined with ApexClass.Name.
- [ ] Both orgs queried via `Promise.all()` — **never sequential**. Verified by grep.
- [ ] LIMIT/OFFSET pagination when result count hits the query cap (addresses High risk: governor limits).
- [ ] Rejects with a user-facing-ready error on 10s timeout.
- [ ] No raw API JSON leaked — always mapped to `PermissionRow[]`.
